# Həll saatı 13. Məsələ oxuma, kənar hallar və debugging

Olimpiadada ballar çox vaxt bilik çatışmazlığından yox, üç şeydən itir: məsələni səhv oxumaq, kənar halı unutmaq və səhvi tapa bilməmək.

## Məsələni necə oxumalı

Hər məsələdə, hər dəfə bu addımları əməl edin :

1. Şərti iki dəfə oxu. Birincidə ümumi mənzərə, ikincidə detallara bax. "Ən azı" vs "düz", "ardıcıl" vs "istənilən", "müsbət" vs "mənfi olmayan" kimi şərtlər məsələnin həllində böyük rol oynayır.
2. Limitlərə bax. n ≤ 100 ➝ n³ keçir; n ≤ 10⁵ ➝ n log n lazımdır; qiymətlər 10⁹-a qədər ➝ long long düşün. Limitlər həllin ipucusudur.
3. Nümunələri əl ilə yoxla. Girişdən çıxışı özünüz alın. Ala bilmirsinizsə, şərti hələ başa düşməmisiniz.
4. Öz nümunəni qur. Ən kiçik hal (n=1), Ən böyük hal, "hamısı bərabərdir" halı.
5. Planı sözlə de. "Oxuyuram, sıralayıram, qonşulara baxıram". 

:::tip[Nümunə cavabı izah edin]
Codeforces nümunələrinin altındakı izahı oxuyun (varsa). İzah yoxdursa, "niyə cavab budur?" sualına özünüz cavab tapın. Bu addım, şərtin gizli qaydalarını ortaya çıxarır.
:::

## Kənar hallar (edge cases)

Kodu göndərməzdən əvvəl bu halları gözdən keçirin:

- n = 1: dövr işləyirmi? qonşu müqayisələri nə edir?
- n = maksimum: TLE? Overflow?
- Elementlərin hamısı bərabər olduqda "ikinci maksimum" və "fərqli qiymətlər" kimi anlayışları olmur.
- Sıfırlar / mənfilər: bölmə? qalıq? mütləq qiymət?
- Artıq sıralı / tərs sıralı giriş
- Cavab 0 və ya boş olan hal: proqram nə çap edir?

## Debugging: səhvi sistemli tapmaq

### 1. Çap ilə izləmə

Aralıq qiymətləri çap etmək:

```cpp
for (int i = 0; i < n; i++) {
    cerr << "i=" << i << " cem=" << cem << '\n';
}
```

`cerr`: xəta axını: yoxlama sistemi onu görmür, ona görə silmədən göndərsəniz belə WA olmaz (amma yavaşlada bilər).

### 2. Kağızda işlətmə

Kodu 3-4 elementlik nümunədə kağızda, sətir-sətir işlədin, dəyişənlərin cədvəlini aparın.

### 3. İkiyə bölmə

Proqram səhv cavab verir, harada? Kodu yarı-yarı bölərək səhvin mənbəyini tapmağa çalışın.

### 4. Səhvlərin növləri

| Verdikt | İlk şübhələr |
|---|---|
| WA | edge case, overfloü, format, `=`/`==`, indeks sürüşməsi |
| TLE | sonsuz dövr, O(n²) böyük n-də, `endl`, sürətli I/O yoxdur |
| RE | massiv sərhədi, 0-a bölmə, böyük lokal massiv |
| Nümunədə düz, testdə səhv | edge case və ya limitlərdə overlow |

## Stress test

Səhvi tapa bilmirsiniz? Kompüter tapsın. Üç proqram yazın:

1. Original həll
2. Brute force həll
3. Generator

:::tip
Balaca test caselər sizə koddakı səhvi tapmaq üçün daha rahat olur.
:::

## Yarış strategiyası (mini)

- Bütün məsələləri əvvəlcə oxuyun, ən asanı həmişə birinci məsələ olmaya bilər.
- Bir məsələyə 30+ dəqiqə ilişdinizsə, başqasına keçin.
- Göndərməzdən əvvəl edge caseləri, nümunələri və tiplərin limitlərini(`int`, `long long`) yoxla.
- WA aldınızda təxminlə düzəltməyin, səbəbini anlayıb düzəldin; kor-koranə göndərişlər etməyin.

## Çalışmalar

**Çalışma 1.** Bu kod "n ədəddən ən böyüyünü" tapmalıdır, amma bəzən səhv işləyir. Səhvi tapın:

```cpp
int maks = 0;
for (int i = 0; i < n; i++) {
    cin >> x;
    if (x > maks) maks = x;
}
```

<details>
<summary>Həlli</summary>

Başlanğıc `maks = 0`: bütün ədədlər mənfidirsə, cavab səhvən 0 çıxır. Düzəliş: ilk elementi başlanğıc götürmək. Bu, "hamısı mənfi" edge case-dir.

</details>

**Çalışma 2.** Bu kod niyə TLE alır?

```cpp
long long n;
cin >> n;                  // n <= 10^18
while (n != 1) {
    if (n % 2 == 0) n /= 2;
    else n -= 1;
    say++;
}
```

<details>
<summary>Həlli</summary>

Alqoritm düzdür və sürətlidir (log addım). Problem yoxdur... əgər n ≥ 1-dirsə. Amma giriş n = 0 verirsə: 0 cütdür, 0/2 = 0 sonsuz dövrlə nəticələnir. Edge case: şərtdə n-in minimumunu yoxlayın.

</details>
